# Ytravel 기술 스택 및 개발 로드맵

## 1. 권장 기술 스택

### 프론트엔드

- Next.js
- TypeScript
- Tailwind CSS
- React Server Components 기반 페이지 구조

선정 이유

- 검색 중심 서비스에 적합
- 서버 렌더링과 SEO 대응이 좋음
- 결과 페이지와 상세 페이지를 빠르게 구성 가능

### 백엔드

- Next.js Route Handler 또는 별도 API 서버
- TypeScript 공용 도메인 모델
- 추후 필요 시 작업 큐 추가

### 데이터 저장

초기

- 파일 기반 모의 데이터 또는 인메모리 캐시

중기

- PostgreSQL
- Redis 캐시

### 수집 계층

- Provider Adapter 구조
- 공급처별 수집 모듈 분리
- 수집 결과를 공통 `RateOffer` 형식으로 정규화

예시 공급처

- Official Website
- Trip.com
- Agoda
- Booking.com

## 2. 추천 아키텍처

### 레이어

- UI Layer
- Search Orchestration Layer
- Provider Adapter Layer
- BRG Evaluation Layer
- Storage / Cache Layer

### 기본 흐름

1. 사용자가 검색 요청 생성
2. 오케스트레이터가 공급처 어댑터 호출
3. 어댑터가 각 가격을 공통 포맷으로 정규화
4. BRG 평가 엔진이 동일 조건과 최저가 계산
5. UI가 결과를 카드/테이블 형태로 렌더링

## 3. 핵심 도메인 설계

### Provider Adapter

공급처별 책임

- 입력 검색 조건 변환
- 가격 수집
- 원본 필드 추출
- 공통 포맷 매핑
- 실패 상태 리턴

### BRG Evaluation Engine

책임

- 공식가 식별
- OTA 최저가 계산
- 동일 조건 판정
- BRG 가능 여부 판정
- 예상 BRG가 계산

### Match Engine

판정 항목

- 객실명 유사도
- 취소 정책 일치
- 조식 포함 여부
- 세금 포함 여부
- 결제 방식

## 4. 단계별 로드맵

### Phase 1. 기획 및 프로토타입

기간

- 1주

산출물

- PRD
- 화면 구조
- 모의 데이터 기반 결과 UI
- BRG 계산 로직 초안

목표

- 제품 방향 고정
- 사용자에게 보여줄 핵심 경험 정의

### Phase 2. MVP 구현

기간

- 2주 ~ 4주

산출물

- 검색 페이지
- 결과 페이지
- 호텔 상세 페이지
- 공통 도메인 모델
- 모의 공급처 어댑터

목표

- 데모 가능한 수준의 웹 서비스 완성

### Phase 3. 제한적 실데이터 연결

기간

- 3주 ~ 6주

산출물

- 일부 공식 사이트 연동
- 일부 OTA 연동
- 캐시와 오류 처리
- 결과 신뢰도 표시 고도화

목표

- 실제 검색 정확도 검증
- 데이터 수집 병목 파악

### Phase 4. 운영 준비

기간

- 2주 ~ 4주

산출물

- 로깅
- 모니터링
- 검색 실패 추적
- 법적/운영 문구 정리

목표

- 외부 사용자 테스트 가능 수준 확보

## 5. 기술 리스크와 대응

### 리스크

- OTA 구조 변경
- 공식 사이트마다 HTML 구조 상이
- 봇 차단 및 지역별 가격 차이
- 통화/세금 처리의 복잡성

### 대응

- 공급처별 어댑터 분리
- 원본 응답 메타데이터 저장
- 캐시 도입
- 비교 결과에 신뢰도와 조건 차이 명시

## 6. 추천 구현 순서

1. Next.js 프로젝트 골격 생성
2. 검색 UI 및 결과 UI 구현
3. 공통 도메인 타입 정의
4. BRG 평가 엔진 구현
5. 모의 공급처 데이터 연결
6. 이후 실제 공급처를 하나씩 붙이기

## 7. 다음 개발 기준선

첫 구현에서는 아래를 반드시 유지한다.

- 공급처별 수집 로직과 평가 로직을 분리
- 결과는 항상 공통 타입으로 정규화
- `확정`보다 `가능성` 표현 사용
- UI에서 조건 차이를 숨기지 않음
