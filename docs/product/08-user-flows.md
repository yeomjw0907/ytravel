# Ytravel 사용자 플로우

## 1. 목적

이 문서는 Ytravel의 대표 사용자 시나리오를 단계별로 정의한다.

목표

- 핵심 사용 흐름을 팀 전체가 동일하게 이해
- 화면 설계와 API 설계의 기준점 확보
- 이탈 가능 구간 사전 파악

## 2. 핵심 사용자 여정

Ytravel의 대표 여정은 아래와 같다.

1. 서비스 유입
2. 호텔 및 일정 검색
3. 비교 결과 확인
4. BRG 가능성 판단
5. 상세 근거 검토
6. 외부 사이트 이동

## 3. 메인 플로우

### Flow A. 첫 방문 후 검색

1. 사용자가 홈 화면에 진입한다.
2. 히어로 영역에서 Ytravel의 목적을 이해한다.
3. 호텔명, 체크인, 체크아웃, 인원, 객실 수를 입력한다.
4. 검색 버튼을 누른다.
5. 시스템이 검색 요청을 생성한다.
6. 결과 페이지로 이동한다.
7. 공홈 가격과 OTA 가격을 정렬된 형태로 확인한다.
8. BRG 상태와 예상 BRG가를 확인한다.
9. 더 확인이 필요하면 호텔 상세 페이지로 이동한다.

### Flow B. 검색 결과에서 바로 행동

1. 사용자가 검색 결과 페이지에 도달한다.
2. BRG 요약 카드에서 `가능성 높음` 또는 `검토 필요`를 확인한다.
3. 공급처 비교 리스트에서 최저가 공급처를 확인한다.
4. 외부 링크를 눌러 해당 공급처 사이트로 이동한다.

### Flow C. 상세 비교 후 판단

1. 사용자가 호텔 상세 페이지로 이동한다.
2. 객실 타입, 취소 정책, 조식 포함 여부, 세금 포함 여부를 비교한다.
3. 공식가와 OTA가 동일 조건인지 확인한다.
4. BRG 진행 가이드를 읽는다.
5. 외부 사이트 링크 또는 공식 사이트 링크를 선택한다.

## 4. 보조 플로우

### Flow D. 검색 수정

1. 사용자가 결과 페이지 상단의 검색 요약 바를 본다.
2. 날짜 또는 인원 조건이 맞지 않음을 인지한다.
3. 검색 수정 버튼을 눌러 조건을 변경한다.
4. 새 조건으로 검색 결과를 다시 본다.

### Flow E. BRG 이해 부족 사용자

1. 사용자가 홈 또는 결과 페이지에서 BRG 개념을 잘 모른다.
2. `BRG란?` 링크를 클릭한다.
3. BRG 설명 페이지에서 개념, 조건, 주의사항을 이해한다.
4. 다시 검색으로 복귀한다.

## 5. 시스템 상태 플로우

### Flow F. 검색 중

1. 사용자가 검색을 실행한다.
2. 시스템은 로딩 상태를 표시한다.
3. 검색 컨텍스트를 상단에 고정한다.
4. 결과 수집 완료 시 결과를 노출한다.

### Flow G. 일부 공급처 실패

1. 시스템이 일부 공급처 가격 수집에 실패한다.
2. 사용자는 나머지 결과는 계속 볼 수 있다.
3. 실패 공급처는 `수집 실패` 상태로 표시한다.
4. 화면은 전체 실패처럼 보이지 않아야 한다.

### Flow H. 결과 없음

1. 검색 조건으로 유효한 결과를 찾지 못한다.
2. 시스템은 `결과 없음` 상태를 노출한다.
3. 사용자는 날짜, 호텔명, 인원 조건을 수정할 수 있다.

## 6. 사용자 의사결정 포인트

사용자는 아래 지점에서 판단을 내린다.

- 이 호텔을 더 볼 가치가 있는가
- OTA 최저가가 충분히 낮은가
- 동일 조건으로 볼 수 있는가
- 지금 바로 외부 사이트로 이동할 것인가
- BRG를 시도할 가치가 있는가

따라서 화면은 항상 아래 질문에 답해야 한다.

- 공식가가 얼마인가
- 가장 싼 OTA가 얼마인가
- 조건이 같은가
- BRG 가능성이 높은가

## 7. 주요 이탈 포인트

- 홈에서 BRG 개념이 어려워 검색을 시작하지 못함
- 결과 페이지에서 조건 비교가 어려움
- 최저가만 보고 동일 조건 여부를 놓침
- 공급처 링크 클릭 전 확신이 부족함

## 8. 개선 장치

- 홈에서 BRG를 3단계로 설명
- 결과 페이지에 BRG 요약 카드 고정
- 조건 차이를 태그로 즉시 표시
- 확정 문구 대신 판단 도움 문구 사용
